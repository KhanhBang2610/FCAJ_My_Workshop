Hello everyone in the AWS Study Group VN community.

Coming from a background of coding traditional Backend processing flows in Java or Node.js, I recently started diving deeper into Cloud AI and RAG (Retrieval-Augmented Generation) architectures. While optimizing a real-world project, I realized a fascinating truth: **AI actually has no idea how to accurately read the words users provide!**

We often hear about giving document files to AI to read and answer questions. But behind the scenes, the system doesn't read Strings the way humans read books. It requires a crucial transformation step called **Embedding**.

Today, I want to dissect the process of using Amazon Bedrock to transform raw data into Vectors – the core of every modern AI system.

![System Workflow](/images/blog4-workflow.jpg)

To build a complete RAG system on AWS, your data goes through a processing pipeline that us Backend developers are very familiar with:

### 1. Chunking
You can't just throw a 100-page PDF into Bedrock and tell it to "Embed this". It's like trying to stuff a whole chicken into your mouth. 
We have to write code to split the document into smaller pieces (chunks) of a few sentences or a paragraph. This chunking helps the AI search for precise information later instead of getting lost in a diluted context.

### 2. Calling Amazon Bedrock
After obtaining the small text chunks, the Backend system calls an API to send that text to Amazon Bedrock (using specialized models like Titan Text Embeddings or Cohere).
Bedrock acts as a "magic black box". It takes a String (e.g., "7-day return policy") and returns a massive array of real numbers (e.g., `[0.012, -0.453, 0.887, ...]`).

### 3. Storing in a Vector Database
Theoretically, you could stuff this number array (Vector) into a JSON or BLOB column in a standard MySQL or PostgreSQL database, and nothing forbids it. The problem lies elsewhere: when you need to perform similarity searches on millions of vectors, traditional databases lack a suitable index mechanism (like HNSW or IVFFlat), so they would have to sequentially scan all the data—which is incredibly slow and resource-intensive.

Therefore, in practice, people usually use specialized Databases for multi-dimensional spaces like Amazon OpenSearch Serverless, or add specialized extensions like `pgvector` to PostgreSQL for ANN (Approximate Nearest Neighbor) indexing, making searches significantly faster than full scans.

Each record now consists of: `[Original Text Chunk] + [Vector Array]`.

---

### How does the AI search when a user asks a question?
This is the most interesting part of the RAG architecture. When a user types in the chatbox: "I want an ABCXYZ outfit", the system performs the following steps:
1. Sends this question to Bedrock to turn it into a Vector (creating a new coordinate on the map).
2. Takes this coordinate into the Vector Database and runs an algorithm called **Cosine Similarity** (measuring the angle between two vectors, not the standard straight-line distance).
3. In plain language: The database uses a protractor to see which documents lie closest to the question's coordinates on that thousand-dimensional map.
4. Extracts the top 5 closest results (the text chunks talking about short-sleeve shirts, shorts) and pushes that text to a Large Language Model (LLM) so it can synthesize the most natural answer.

All of this happens in just a few hundred milliseconds!

---

### Why AWS Bedrock and not self-hosted Models?
From a system architecture perspective, self-hosting open-source Embedding models on EC2 is a nightmare. You have to worry about GPU setups, environment configurations, and especially solving the Auto-scaling puzzle when request traffic spikes.

With Amazon Bedrock, the infrastructure is almost entirely managed by AWS (serverless). You just configure the SDK in Java or Node.js, call the API, and AWS handles scaling the underlying infrastructure within your account limits. You pay for what you use. This allows developers to focus entirely on building business logic instead of becoming infrastructure "plumbers."

---

### "Battle Scars" from integrating Bedrock Embedding
- **Rate Limit (Throttling) Errors:** When writing a script to read thousands of document pages and continuously calling the Bedrock API to convert them to Vectors, it's very easy to hit API limits (account/region quotas). Always implement a Retry mechanism (retry after a few seconds) or throw them into a queue (Amazon SQS) to call them gradually.
- **Chunking strategy is a matter of life and death:** If you cut text too short, the AI loses context (it doesn't understand what is being talked about). Cut too long, and information gets tangled during searches. The rule of thumb is to let text chunks "overlap" slightly.
- **Vector DBs cost money:** Storing Vectors consumes much more RAM than plain text. Calculate costs carefully when choosing a storage service.
- **API Call costs need calculation:** Besides Vector DB storage costs, calling Bedrock to mass-embed documents (especially re-embedding due to a change in chunking strategy) also incurs fees per processed token. With large datasets, this adds up significantly, so think twice before running full document batches repeatedly.

### Conclusion
Amazon Bedrock is not just a hub for fancy models like Claude or Llama; it also provides an incredibly powerful Embedding tool to breathe life into raw data.

Mastering the **Chunking → Embedding → Vector Search** process is the key for a Backend Developer to confidently build awesome AI features without needing to be a math or data science expert.
